# Архитектура

## Подход

Task Hub проектируется как модульный монолит.

На текущем этапе это один backend-сервис, внутри которого выделены независимые слои:

```text
HTTP API
  -> Controller
  -> Service
  -> Repository
  -> Database
```

Такой подход проще микросервисной архитектуры, но сохраняет разделение ответственности и позволяет развивать систему без смешивания HTTP-логики, бизнес-правил и доступа к данным.

## Основные слои

### HTTP API

Отвечает за:

- регистрацию маршрутов;
- приём HTTP-запросов;
- передачу данных в контроллеры;
- возврат HTTP-ответов.

### Controller

Отвечает за:

- получение уже провалидированных данных;
- вызов сервисного слоя;
- выбор HTTP-статуса ответа.

Контроллер не содержит бизнес-правил и прямых запросов к базе данных.

### Service

Отвечает за бизнес-логику:

- создание задачи;
- проверку бизнес-ограничений;
- обновление задачи;
- soft-delete;
- формирование доменных событий.

### Repository

Отвечает за работу с базой данных.

Repository скрывает детали Prisma от сервисного слоя.

### Database

Для хранения данных используется PostgreSQL.

Доступ к базе выполняется через Prisma.

## Модуль задач

Основной модуль системы:

```text
src/modules/tasks
```

Планируемые файлы модуля:

```text
task.routes.ts
task.controller.ts
task.service.ts
task.repository.ts
task.schemas.ts
task.mapper.ts
task.types.ts
task.events.ts
```

## Интеграционный слой

Интеграции рассматриваются как отдельный слой, который не должен смешиваться с основной логикой задач.

Базовая идея:

```text
Task Service
  -> Domain Event
  -> Integration Event Publisher
  -> External System Adapter
```

На первом этапе внешний обмен не реализуется. В коде уже присутствует точка расширения:

```text
TaskService
  -> TaskEvent
  -> IntegrationEventPublisher
  -> NoopIntegrationEventPublisher
```

`TaskService` формирует события `task.created`, `task.updated`,
`task.completed` и `task.deleted`. Текущий publisher намеренно ничего не
отправляет. После согласования контрактов его можно заменить адаптером, не
добавляя интеграционную логику в контроллеры или репозиторий.

## Будущие входящие интеграции

Внешние системы потенциально смогут:

- создавать задачи;
- изменять задачи;
- завершать задачи;
- связывать свои объекты с задачами Task Hub.

Для этого могут использоваться:

- основной REST API;
- отдельные интеграционные эндпоинты;
- idempotency key для защиты от дублей;
- таблица связей между задачами и внешними объектами.

## Будущие исходящие интеграции

Task Hub потенциально сможет уведомлять внешние системы о событиях:

- задача создана;
- задача изменена;
- задача выполнена;
- задача удалена.

Способ доставки зависит от возможностей конкретной внешней системы:

- REST API;
- webhook;
- очередь сообщений;
- периодическая синхронизация.

## Почему не добавляется брокер сообщений сразу

Брокер сообщений имеет смысл, если есть конкретные сценарии асинхронной доставки, несколько потребителей событий или требования к повторной отправке.

Пока таких контрактов нет, брокер не добавляется в первую версию. Вместо этого сохраняется архитектурная точка расширения, чтобы позже можно было добавить outbox, очередь или адаптеры без переписывания доменной логики задач.

## Граница первой версии

В первую версию входит:

- REST API задач;
- PostgreSQL;
- Prisma;
- валидация;
- единый формат ошибок;
- soft-delete;
- документация;
- OpenAPI;
- Docker.
- доменные события задач;
- интерфейс integration publisher и no-op реализация.

В первую версию не входит:

- пользователи;
- авторизация;
- роли;
- реальные интеграции с внешними системами;
- брокер сообщений;
- восстановление удалённых задач.
